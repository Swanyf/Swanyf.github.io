线程池由浅入深

作者 swan 日期 2019-06-13
java
线程池由浅入深

new Thread

弊端

1.new Thread 每次都会创建对象，性能差
2.线程缺乏统一管理，可能无限制的新建线程，相互竞争，有可能占用过多系统资源导致死机或 OOM
3.缺少更多功能，如更多执行、定期执行、线程中断，方法使用复杂

线程池

优点

1.重用存在的线程，减少对象创建、消亡的开销，性能佳
2.可有效控制最大并发线程数，提高系统资源利用率，同时可以避免过多资源竞争，避免阻塞
3.提供定时执行、定期执行、单线程、并发数控制等功能

ThreadPoolExecutor

public ThreadPoolExecutor(int corePoolSize,
                          int maximumPoolSize,
                          long keepAliveTime,
                          TimeUnit unit,
                          BlockingQueue<Runnable> workQueue,
                          ThreadFactory threadFactory,
                          RejectedExecutionHandler handler) {
    if (corePoolSize < 0 ||
        maximumPoolSize <= 0 ||
        maximumPoolSize < corePoolSize ||
        keepAliveTime < 0)
        throw new IllegalArgumentException();
    if (workQueue == null || threadFactory == null || handler == null)
        throw new NullPointerException();
    this.acc = System.getSecurityManager() == null ?
            null :
            AccessController.getContext();
    this.corePoolSize = corePoolSize;
    this.maximumPoolSize = maximumPoolSize;
    this.workQueue = workQueue;
    this.keepAliveTime = unit.toNanos(keepAliveTime);
    this.threadFactory = threadFactory;
    this.handler = handler;
}

corePoolSize: 核心线程数量
maximumPoolSize : 最大线程数
workQueue : 阻塞队列，存储等待执行的恩物，很重要，会对线程池运行过程产生重大影响

如果运行的线程数少于 corePoolSize 时，直接创建新的线程来处理任务，即使线程池中其他线程是空闲的
如果线程池中的线程数量大于等于 corePoolSize，小于 maximumPoolSize最大线程数，则只有当 workQueue 满的时候，才创建新的线程去处理任务
如果设置的 corePoolSize 和 maximumPoolSize 相同时，此时如果有线程提交，如果 workQueue还没满，就把请求放到 workqueue 中，等待有空闲的线程去取出任务去处理
如果运行的线程数大于 maximumPoolSize 最大线程数，且 workQueue 也已经满了会通过一个拒绝策略的参数，来指定策略去处理任务
如果想降低系统资源的消耗，包括cpu的使用率、操作系统资源的消耗，可以设置一个较大的队列容量，和较小的线程池容量，这样会降低线程处理任务的吞吐量。
如果我们提交的任务经常发生阻塞，可以考虑调用设置线程最大数的方法，来重新设置线程池容量
如果队列的容量设置较小，需要把线程池容量设置的大一点，cpu的使用率会相对高一些
如果线程池容量设置的过大，在提交任务过多的情况下，并发量会增加，这时线程之间的调度就是需要考虑的一个问题，这样会降低处理任务的吞吐量

keepAliveTime : 线程没有任务执行时最多保持多久时间停止
当线程池中的数量大于u corePoolSize 时，如果这时没有新的任务提交，核心线程不会立即销毁而是等待，直到等待的时间超过 keepAliveTime
unit : keepAliveTime 的时间单位
threadFactory : 线程工厂，用来创建线程，会有一个默认的线程工厂来创建线程。
使用默认的线程工厂来创建线程时，会使新创建的线程具有相同的优先级，并且是非守护的线程也设置的线程的名称
rejectHandler : 当拒绝处理任务时的策略
当 workQueue 阻塞队列满了且没有空闲的线程池，这时需要拒绝提交任务我们就需要采取一种策略来处理任务。线程池提供了四种处理方式：

线程池实例的几种状态

Running


判断接受新提交的任务，也能处理阻塞队列中的任务

SHUTDOWN


关闭状态，不能再接受新提交的任务，但是可以继续处理阻塞队列中已经保存的任务，在线程池处理Running状态时，调用Shutdown会进入该状态

Stop


不能接受新的任务，也不能处理队列中的任务，会中断正在处理任务的线程，在线程池调用处于Running、Shutdown状态如果调用ShutdownNow会进入该状态

Tidying


如果所有的任务都已经终止了线程数为0，会进入该状态。之后调用 terminated() 方法会进入 Terminared状态

线程池提供的方法

execute()


提交任务，交给线程池执行

submit()


提交任务，能够返回执行结果 execute + Future

shutdown()


关闭线程池，登台任务都执行完

shutdownNow()


关闭线程池，不等待任务执行完
